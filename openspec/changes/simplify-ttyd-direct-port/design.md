## 上下文

当前 MR 容器的 ttyd 终端通过 `http-proxy-middleware` 反向代理访问。代理层需要处理 HTTP 请求转发、WebSocket 升级、路径重写三个维度，且与 Express 的路由挂载机制存在 `req.url` 截断冲突，导致 `router` 回调无法正确解析目标地址。

测试容器已采用端口映射 + 直连方案（`HostPort: "0"` 随机映射），运行稳定。系统部署在单机环境，所有容器在同一台宿主机上。

## 目标 / 非目标

**目标：**
- MR 容器的 ttyd 采用与测试容器一致的端口映射直连方案
- 删除 MR 容器相关的反向代理代码
- 前端 iframe 直连宿主机 ttyd 端口

**非目标：**
- 不改变测试容器的终端访问方式
- 不改变容器生命周期管理逻辑
- 不支持多机分布式部署场景

## 决策

### 决策 1：MR 容器 ttyd 端口映射到宿主机随机端口

**选择**：创建 MR 容器时，将 7681 端口映射到宿主机随机端口（`HostPort: "0"`）

**替代方案**：
- A. 修复现有代理逻辑（改用 `req.originalUrl`）— 治标不治本，代理层复杂度仍在
- B. 使用 Nginx 做反向代理 — 引入额外依赖，增加运维复杂度

**理由**：测试容器已验证此方案可行，代码改动最小，彻底消除代理层问题。

### 决策 2：前端通过 status API 获取 ttyd 端口

**选择**：status API 在 `ports` 字段中返回 7681 对应的宿主机端口，前端用 `location.hostname` + 该端口构建 iframe URL

**替代方案**：
- A. 新增专用 API 返回 ttyd URL — 过度设计，ports 字段已包含所需信息

**理由**：复用现有 `ports` 字段，零新增 API。

### 决策 3：删除 MR 代理代码，保留测试容器代理

**选择**：从 `ttyd-proxy.ts` 中删除 MR 相关的 `createProxyMiddleware`、Express 中间件和 WebSocket upgrade handler。保留测试容器部分不变。

**理由**：测试容器的代理逻辑独立且稳定，无需一起改动。MR 容器改为直连后，其代理代码成为死代码。

## 风险 / 权衡

- **[端口暴露]** ttyd 端口映射到宿主机后，理论上任何能访问宿主机的人都可以直连 ttyd → 当前系统本身无认证机制，代理方案也不提供额外安全性，风险不变
- **[跨域]** iframe 加载不同端口的页面，同源策略限制 JS 跨 frame 通信 → ttyd 是独立页面，不需要与父页面 JS 通信，无影响
- **[端口耗尽]** 大量容器同时运行可能耗尽可用端口 → 已有 `maxContainers` 配置限制并发数，风险可控
